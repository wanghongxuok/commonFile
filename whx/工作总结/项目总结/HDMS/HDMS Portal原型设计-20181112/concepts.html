Catalogs

NavInfo stores data in catalogs. A catalog is a collection of segmented data consisting of different layers which used to store different kinds of data. For example, a catalog used to store map data may contain layers of lane, obstacle, etc. Catalogs can not only store geospatial data but also other data, such Campaign Jobs and Sensor data collected from the vehicles. Usually, data of different business types are organized into different catalogs for the convenience of data management.

Layers

A layer is a collection of data with a specific data type, functional attributes, and structure. For a catalog of map data, different aspects of the map's content are organized as layers, for example, the boundary markers separating different administrative areas are stored in the Landmark layer. Multiple layers complement each other in space and content to form a complete digital map.

Versioned Layers

Versioned layers can be used to store data that needs to be updated at low frequencies and must remain logically consistent with other layers in the catalog. When you want to update the versioned layers in a catalog, all the layers related to the update content(and partitions within the layer) must be updated in a batch so that they can be versioned together. For example, when a new road needs to be added to the lane layer, all partition ranges where the road is located need to be updated. Other layers associated with the road, such as points of interest and speed limits, also need to be updated synchronously.


In order to maintain consistency between data, when updating the versioned layer, it is of great importance for you to upload all the relevant data and publish them in a batch. New version of data can only be obtained if all the data is published successfully. Once a new version is released, data within the version can not be added, deleted, or changed.


In the same NDS Spec, we update the data incrementally instead of replacing the entire layer. When data in a versioned layer is updated, the catalog version is incremented by one. It should be noted that only the version of updated partitions and layers is changed to match the new catalog version, layers and partitions that are not updated keep their existing version numbers.

Dynamic Layers

Dynamic layers can be used to store dynamic traffic information with high frequency of change, such as traffic accident, speed limit and warning signs. All values that can be changed dynamically shall have a unique and stable identifier across all HD Map Releases. To reference dynamic data from any supplier onto the lane model of HD map, it is recommended that all the dynamic data matches with the location reference we provide. We will compile all the data according to the matching results and provide unified service with other layers.

OEM Layers

There are three types of layers: versioned layers, dynamic layers and OEM layers.

OEM layers can be used to store OEM data attributes from customer or its agents, it can be applied to assist in highly automated driving, such as Where/When /Why enabling and disabling highly automated driving system. According to the update requirements and the update frequency, there are three types of layers: Semi-permanent layer, Semi-Dynamic Layer, Dynamic Layers.

Partitions

 A partition is the smallest unit of map data storage in the system which used to store any binary data. Multiple partitions form a layer and multiple layers form a catalog. Both layers and catalogs are the logical organization of multiple partitions, which makes the catalog (with layer and partition) a key-key-value store.

Sensor Data

Campaign Jobs

Sensor data defined by NavInfo is used to define crowdsourcing data collected from vehicles for map updates. It usually includes information as follows.

• Information about data for map updates, such as vehicle postion, lane boundary, lane markings.

• Information about the vehicle status, such as in-car temperature, state of door.

• Information about the vehicle dynamics, such as angle of steering wheel, position of brake pedal.

• Information about the environment, such as weather, external air temperature.

Campaign jobs defined by NavInfo are used to define a series of crowdsourcing data acquisition tasks to be performed. Based on different data acquisition strategies, the job list in a campaign job may include one or more jobs. For the convenience communication with the Backend, a unified definition of the campaign jobs is made by NavInfo, it usually includes:

• Which data has to be collected

• Where the data is collected

• When the data is collected

• How long and how frequent the data is collected

• The (complex) trigger conditions when the data collection starts

• Pre- and post trigger count of Position Estimates

• How the data is aggregated

• Size of the individual Sensor Data Messages and Segmentation strategy

• Prioritizing and discarding Sensor Data packets

Catalogs have the configuration information that defines their content, including the catalog 's geographic coverage, version, layers, timestamp, and so on.

A catalog is a combination of three types of layer:

• Versioned layers which used to store data that does not change frequently, such as lane and routing.

• Dynamic layers which used to store data that changes frequently, such as traffic incidents and speed limit.

• OEM layers which is used to store data produced by third party.

For different types of data, we provide two partitioning schemes: Generic partitioning and Tile partitioning.

Generic partitioning is an internal custom organization form of partitions and is recommended used to other data other than map data.

Tile partitioning defined by NavInfo is to divide geographical range into blocks according to certain grid spacing and is recommended used to map data. A partition represents a particular area of geographical space. Each partition has a unique encoding called partition ID and does not overlap geographically. Multiple partitions that are contiguous in geographical space and logically consistent form a layer.
